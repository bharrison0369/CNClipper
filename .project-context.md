## Development Plan

### Phase 1 â€” Core CNC Primitives (implement first)
These are the universal building blocks every other module depends on. Grounded in RS274/NGC standard G/M-codes.

- [ ] **`cnc_wcs`** â€” Work Coordinate Systems (G54-G59, G59.1-G59.3, G10 L2/L20, G53, G92)
  - 9 persistent coordinate systems with G10 setting and G53 machine-coord moves
  - Use Klipper's `SET_GCODE_OFFSET` internally (proven by noobydp/CNC-Macros) + `save_variables` for persistence
  - G92 temporary offset layer on top of WCS
  - Validate: mill fixture work, waterjet sheet alignment, laser bed origin â€” same API, different workflows
- [ ] **`cnc_io`** â€” Synchronized and immediate digital/analog I/O (M62-M68, M66)
  - Two-tier model from LinuxCNC/grblHAL: motion-synchronized (M62/M63/M67) vs immediate (M64/M65/M68)
  - M66 input wait with timeout and edge detection (RISE/FALL/HIGH/LOW)
  - This is the primitive that output_sequencer and interlock compose on top of
  - Validate: waterjet valve on/off, plasma arc-OK wait, laser shutter trigger â€” same M-codes, different pins
- [ ] **`cnc_tool_output`** â€” Universal tool activation (M3/M4/M5 reimagined)
  - Replaces Klipper's PWM-only spindle with grblHAL-style capability model
  - Tool types defined by interface, not by name: capabilities bitmap (variable_power, direction, at_speed_feedback, motion_gated, requires_startup_sequence)
  - Same M3/M4/M5 commands drive a PWM spindle, digital solenoid (waterjet valve), laser, or VFD â€” config determines behavior
  - M4 = dynamic power mode (power scales with velocity) for lasers, following grblHAL $32 pattern
  - Configurable startup_delay and shutdown_delay (spindle spin-up, laser gas pre-flow)
  - Validate: spindle (PWM + at-speed), waterjet valve (digital on/off), laser (PWM + motion-gated), plasma torch (digital + arc-OK feedback) â€” one module, four configs
- [ ] **`cnc_dwell`** â€” G4 dwell (timed motion pause, no output side effects)
  - Clean implementation of RS274/NGC G4 â€” axis pause only, all outputs maintain state
  - Klipper may already handle this adequately; evaluate before implementing

### Phase 2 â€” Composed Safety & Sequencing (builds on Phase 1)
These compose the Phase 1 primitives into higher-level patterns.

- [ ] **`cnc_output_sequencer`** â€” Parameterized timed output sequences
  - Composes cnc_io + cnc_dwell + cnc_tool_output into multi-step startup/shutdown sequences
  - Config-driven steps: [{output: "valve", action: "on"}, {dwell: 0.5}, {wait_input: "pressure_ok", timeout: 3.0}, {begin_motion: true}]
  - NOT a pierce module, NOT a pre-flow module â€” a universal sequencer that any machine type configures for its needs
  - Validate: waterjet pierce (valve on â†’ dwell â†’ cut), plasma arc start (torch fire â†’ wait arc-OK with retry â†’ dwell â†’ THC enable), laser pre-flow (gas on â†’ dwell â†’ laser on â†’ cut), mill spin-up (spindle on â†’ wait at-speed â†’ plunge)
- [ ] **`cnc_interlock`** â€” Configurable safety input gating
  - Composes cnc_io input monitoring into safety rules: "check input X before allowing output Y"
  - Configurable response per interlock: pause, abort, alarm
  - Configurable resume sequence (re-energize, delay, restore)
  - Fail-safe: deny if sensor state unknown (existing STANDARDS.md requirement)
  - Validate: door switch (pause + disable tool), coolant flow sensor (abort if lost mid-cut), chuck clamp (block spindle start), arc-OK (abort if lost during cut)
- [ ] **`cnc_feed_hold`** â€” Motion pause with controlled deceleration
  - Real-time feed hold that decelerates within accel limits and preserves path position
  - NOT Klipper's PAUSE macro (which is for filament changes)
  - Configurable hold behavior: disable tool output (laser safety), keep spindle running (mill), etc.
  - Resume with optional re-energize delay
  - **BLOCKER RISK:** May require Klipper core motion planner hooks â€” prototype early to determine feasibility in extras/ only
- [ ] **`cnc_overrides`** â€” Runtime feed rate and tool power overrides
  - Feed rate override (M50), spindle/power override (M51)
  - Real-time adjustment without replanning â€” applies scaling factor to active motion

## Known Issues / Blockers

- **Feed hold is the hardest problem.** grblHAL implements feed hold as a real-time motion interrupt with controlled deceleration, path position preservation, and planner buffer retention. Klipper's PAUSE is a macro designed for filament changes â€” fundamentally different. A true CNC feed hold needs motion planner integration that may not be possible in extras/ alone. klipper-plasma solved a related problem (pierce dwell) by freezing Klipper's step scheduler at the MCU level (`time_freeze`/`time_unfreeze`), but that's an invasive core mod. **Action:** Prototype early in Phase 2 to determine if extras/-only is viable or if a Klipper core PR is unavoidable.
- **Motion-synchronized I/O (M62/M63) needs planner integration.** LinuxCNC and grblHAL queue output changes to fire at the start of the next motion command. This requires inserting output events into the motion planner queue. Klipper's `output_pin` fires immediately. Immediate I/O (M64/M65) is straightforward; synchronized I/O may need creative approaches (queueing output changes in the move handler).
- **Klipper's M3/M4/M5 is PWM-only** â€” stock `[output_pin]`/`[pwm_tool]` maps spindle commands to PWM duty cycle. cnc_tool_output will need to override this with a capability-based model that supports digital on/off (waterjet), PWM (spindle), and motion-gated (laser) modes. Using `rename_existing` to intercept is the proven Klipper pattern (used by KCNC).
- **Cutter compensation (G41/G42) requires look-ahead** â€” low priority, CAM handles kerf comp for waterjet/plasma/laser. Defer indefinitely.
- **Some real-time operations may eventually need MCU firmware.** klipper-plasma demonstrated that plasma THC needs microsecond-level response that can't run on the host. Laser PPI (pulses per inch) similarly needs MCU timing. Plan for an optional `cnc_mcu` firmware component in the future, but the core Python modules come first.

## Decisions & Context

### Why a module library, not a fork?
Klipper's `klippy/extras/` system is the intended extension mechanism. Modules only load when referenced in `printer.cfg` â€” zero bloat for users who don't need CNC features. This approach:
- Works with stock Klipper (no fork to maintain)
- Distributable via Moonraker's update manager
- Can be upstreamed to Klipper if maintainer accepts PRs
- Won't go stale when Klipper updates (no merge conflicts)

### Why one repo, not one per module?
Modules are interdependent (waterjet depends on output_sequencer depends on interlock). One repo = one version = no compatibility matrix. This matches how klippy/extras/ itself works â€” all in one tree, load only what's configured.

### Architecture: 4-tier module system
1. **core/** â€” Universal CNC primitives (WCS, I/O, tool output, dwell, feed hold, overrides)
2. **common/** â€” Composed patterns most machines need (output sequencer, interlock, probing, tool table, coolant)
3. **machine/** â€” Shared by 2-3 machine classes (THC, gap servo, tangential control)
4. **machines/** â€” Machine-specific config + thin logic (waterjet, plasma, mill, EDM, lathe, PnP)

### Key design principle: machine-agnostic abstractions (see STANDARDS.md)
Every module must be grounded in universal CNC concepts, not machine-specific implementations. A "pierce cycle" is really just timed output sequencing â€” the same pattern as laser pre-flow and spindle spin-up. If a module can't serve 3+ machine types through configuration alone, the abstraction is wrong. This principle is codified in STANDARDS.md and applies to all agents.

### Key design insight: composition hierarchy
Phase 1 modules are primitives (I/O, tool output, dwell). Phase 2 modules compose them (output_sequencer = I/O + dwell + tool_output; interlock = I/O input monitoring + response rules). Machine modules in Phase 3-4 configure the composed modules, adding only truly unique logic.

### Machine types analyzed
Full requirements analysis completed for: waterjet, plasma, CO2 laser, fiber laser, diode laser, mill/router, wire EDM, sinker EDM, lathe, drag knife, foam cutter, pick-and-place. Key validation targets for API design: waterjet (digital solenoid), plasma (digital + arc-OK feedback + THC), laser (PWM + motion-gated), mill (PWM + at-speed + VFD). If the Phase 1-2 APIs can express all four through configuration, the architecture is correct.

### Research: existing Klipper CNC projects (2026-04-11)
- **klipper-plasma (proto3):** 22 stars, abandoned 2020. Full Klipper fork with MCU-level THC and time_freeze for pierce. Proves plasma needs MCU-level timing. Cautionary tale: fork = abandonment.
- **KCNC (vladbabii):** 138 stars, stale since 2022. Pure macros for WCS (G54-G59). Works but hits macro ceiling: no persistence, Jinja2 overhead on every move, stubs for spindle/coolant.
- **noobydp/CNC-Macros:** 15 stars, active 2025. Improved KCNC approach â€” uses SET_GCODE_OFFSET + save_variables. Key lesson: leverage Klipper builtins instead of reimplementing offset math.
- **klipper-for-cnc (naikymen):** 148 stars, active 2026. ABC rotary axes via core fork. Most comprehensive multi-axis support. Heavy merge burden acknowledged. No WCS or spindle abstraction.
- **klipper4cnc (klipper4cnc-coder):** 2 stars, brand new Feb 2026. Full CNC pipeline in extras/ (parser â†’ interpreter â†’ planner â†’ executor). Ambitious, stays in extras/, but barely tested. Interesting architecture to watch.
- **grblHAL:** 477 stars, very active. Gold standard for CNC controller architecture. Spindle defined by capability bitmap + function pointers. Plugin architecture via HAL function pointer replacement. Plasma THC as plugin, not core. **Primary reference for cnc_tool_output design.**

### Research: CNC standard G/M-code mapping (2026-04-11)
- **G4 dwell:** Pure axis pause, no output side effects. Universal primitive.
- **M62-M68:** Two-tier I/O â€” motion-synchronized (M62/M63/M67) vs immediate (M64/M65/M68). Both LinuxCNC and grblHAL implement this. Key universal abstraction for cnc_io.
- **M66:** Input wait with timeout and edge modes (IMMEDIATE/RISE/FALL/HIGH/LOW). Result in parameter #5399. Essential for arc-OK, pressure-OK, at-speed signals.
- **M3/M4/M5:** Universal "activate tool" â€” meaning depends on tool type. grblHAL: SpindleType_PWM, SpindleType_Basic (on/off), SpindleType_VFD, SpindleType_Solenoid. M4 = dynamic power mode (laser velocity scaling).
- **Feed hold:** Real-time deceleration preserving path position. Fundamentally different from Klipper PAUSE. grblHAL: separate state machine with configurable spindle/coolant behavior during hold and resume.
- **Canned cycles (G81-G89):** Mill/drill-specific, but the pattern (parameterized multi-step operation) is universal. Could generalize as a cycle framework later.
