## Decisions & Context

### Repo Boundary

CNClipper owns reusable CNC capabilities shared across multiple machine classes.

The `Wazer` repo owns:

- Wazer-specific `printer.cfg`
- hardware mapping
- validation notes
- process-specific workflows

This repo must not absorb Wazer-specific implementation details just because Wazer is the first proving target.

### Addon-First Strategy

Current implementation strategy is:

1. Try `klippy/extras/` modules first.
2. If the addon layer cannot express a feature cleanly, identify the missing hooks.
3. Only then consider a narrow Klipper patch or fork.

The current plan is explicitly not to rebuild Klipper first and then re-add 3D printing as an addon.

### Why CNClipper Exists

The goal is for:

- stock Klipper
- plus CNClipper
- plus a machine-specific repo

to provide a complete path to running a machine without duplicating shared controller logic in every machine repo.

CNClipper therefore focuses on reusable controller semantics, not machine recipes.

### Shared Scope Versus Machine Scope

Shared scope includes capabilities that are:

- not fully covered by stock Klipper, or only covered in a printer-shaped way
- useful across multiple machine classes
- controller-level semantics instead of one-machine workflows

Machine-specific scope includes sequences and workflows that can be composed from shared primitives.

Example: a waterjet pierce sequence should usually stay in the machine repo if it can be built from shared tool control, I/O, and dwell primitives.

### Validation Sequence

Validation should proceed in this order:

1. Existing working Klipper 3D printers for safe motion and API experiments.
2. Wazer as the first real CNC proving target.
3. Additional machine classes later.

Wazer is the first proving target, not the center of the architecture.

### Machine Classes In View

The long-term design space includes:

- waterjet
- laser cutter / engraver
- plasma cutter
- drag knife
- oxy-fuel
- nibbler
- mill / router
- lathe
- wire EDM
- sinker EDM
- hybrid additive / subtractive
- 6-axis robot
- welding
- pick and place
- CMM
- wire bending

CNClipper does not need to implement every class directly. It needs to capture the capabilities shared across those classes.

### UI Scope

UI work is out of scope for CNClipper. Browser-side and operator interface work is a future project.

### Fork Discussion

A machine-agnostic Klipper-derived platform may be valuable long-term, especially given Klipper's host-plus-MCU split and multi-MCU architecture.

That said, current implementation work should not assume a broad fork. Fork pressure should come from failed addon-first feasibility attempts on specific capabilities, not from architectural preference alone.
