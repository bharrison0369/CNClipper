## Architecture

### Design Principle: Machine-Agnostic Abstractions

**All modules must be grounded in universal CNC concepts, not machine-specific implementations.**

When designing a module, ask: "What is this operation in standard CNC terms?" If the answer maps to an existing G/M-code concept or a pattern shared across machine types, the module must use that universal framing, not a machine-specific name or behavior.

Example: a waterjet pierce cycle, a plasma arc-start sequence, a laser gas pre-flow, and a mill spindle spin-up are all versions of the same pattern: timed output sequencing with optional input gating. The correct abstraction is a shared sequencing primitive, not a `waterjet_pierce` one-off.

**Validation rule:** Before finalizing any module API, describe at least three different machine types using that API. If the API cannot express all three cleanly without machine-specific flags or special cases, the abstraction is wrong.

**Reference CNC standards for naming and behavior:**
- RS274/NGC for core G/M-code concepts
- LinuxCNC for CNC controller semantics and I/O models
- grblHAL for reusable controller capability patterns

### Design Principle: Addon-First Integration

**CNClipper must prefer addon-style Klipper integration before considering core patches or a fork.**

Default implementation order:

1. Implement the feature as a `klippy/extras/` module.
2. If the addon layer cannot express the behavior cleanly, identify the missing Klipper hooks.
3. Only then consider a narrow Klipper patch or fork, and keep that surface as small as possible.

A broad fork that attempts to redesign Klipper into a machine-agnostic controller platform is out of scope for current implementation work unless the user explicitly approves that direction after feasibility work.

### Repo Boundaries

- `CNClipper` owns reusable CNC capabilities shared across multiple machine classes.
- Machine-specific configuration, validation, pin maps, and process workflows belong in machine repos.
- For the Wazer proving target, Wazer-specific logic, `printer.cfg`, hardware notes, and validation remain in the `Wazer` repo.
- UI work is out of scope for `CNClipper`.

### Module Tiers

Current working tier model:

1. `core/` - universal CNC primitives
2. `common/` - composed patterns reused across many machine classes
3. `machine/` - features shared by a smaller subset of machine classes
4. `machines/` - thin machine-specific integration

This tier model is provisional during Phase 0 and may be revised if it proves more confusing than useful.

### Module Design Rules

- Each module should be a standalone `klippy/extras/` Python file unless a clear reason exists to organize shared helper code separately.
- Modules must only load when referenced in config.
- Modules must not modify Klipper core files unless the user has explicitly approved a core patch path.
- Cross-module dependencies must be explicit through printer object lookups.
- All registered G-code or M-code commands must be documented in module docstrings and user-facing docs.
- Shared modules should own controller semantics, not machine recipes.

### Klipper API Conventions

- Follow Klipper coding style and object patterns where practical.
- Use `config.get*()` methods for config parsing during module construction.
- Register commands through the Klipper gcode object.
- Use event handlers for late binding and startup validation.
- Use Klipper pin abstractions instead of direct GPIO access.

## Naming Conventions

- **Module files:** `cnc_<feature>.py` for shared modules, `<machine_type>.py` for thin machine integrations
- **Config sections:** `[cnc_<feature>]` for shared modules, `[<machine_type>]` for machine-specific integrations
- **G-code commands:** use standard G/M-codes where they already exist; prefix truly custom commands with a clear namespace
- **Python classes:** PascalCase
- **Config keys:** snake_case

## Safety / Warnings

- Safety-critical modules must fail safe when sensor or state information is missing or ambiguous.
- Output sequencing must complete or abort cleanly and must not leave dangerous outputs in partial states.
- Machine-specific safety layers must document what happens if a sensor fails or reports stale data.
- Never assume hardware state without confirmation.
- Dangerous process outputs should integrate with Klipper watchdog/disconnect behavior where possible.
