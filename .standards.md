## Architecture

### Design Principle: Machine-Agnostic Abstractions

**All modules must be grounded in universal CNC concepts, not machine-specific implementations.**

When designing a module, ask: "What is this operation in standard CNC terms?" If the answer maps to an existing G/M-code concept or a pattern shared across machine types, the module must use that universal framing â€” not a machine-specific name or behavior.

**Example:** A waterjet pierce cycle, a plasma arc-start sequence, a laser gas pre-flow, and a mill spindle spin-up are all the same universal pattern: *timed output sequencing with optional input gating*. The correct abstraction is `cnc_output_sequencer` (compose dwell + output control + input wait), not `waterjet_pierce` or `plasma_arc_start`. If you build a pierce module that only works for waterjet, you've already failed â€” it should be viable for any machine type that needs a timed startup sequence.

**Validation rule:** Before finalizing any module API, describe at least 3 different machine types' use of that module using the same API. If the API can't express all 3 without machine-specific flags or special cases, the abstraction is wrong â€” go more universal.

**Reference CNC standards for naming and behavior:**
- G4 (dwell), M62-M68 (synchronized/immediate I/O), M66 (input wait) from RS274/NGC
- grblHAL's spindle capability model (tools defined by interface, not type)
- LinuxCNC's two-tier output model (motion-synchronized vs. immediate)

### Module Tiers

1. **core/** â€” Universal CNC primitives every machine type needs (WCS, I/O control, tool output, dwell, feed hold, overrides)
2. **common/** â€” Composed patterns most machines need, configured per type (output sequencer, interlock, probing, tool table, coolant)
3. **machine/** â€” Features shared by 2-3 machine classes (THC, gap servo, tangential control)
4. **machines/** â€” Machine-specific config + thin unique logic (waterjet, plasma, mill, EDM, lathe, etc.)

### Module Design Rules

- Each module is a standalone klippy/extras/ Python file
- Modules only load when referenced in printer.cfg (zero bloat for unused features)
- Modules MUST NOT modify Klipper core files â€” use only public klippy APIs
- Cross-module dependencies must be explicit (use `printer.lookup_object()`)
- All G-code/M-code registrations must be documented in module docstrings

### Klipper API Conventions

- Follow Klipper's existing coding style (see `klippy/extras/` examples)
- Use `config.get()`, `config.getfloat()`, `config.getint()` for config parsing
- Register G-code commands via `self.gcode.register_command()`
- Use `self.printer.register_event_handler()` for event hooks
- Pin access via `ppins.setup_pin()` â€” never direct GPIO

## Naming Conventions

- **Module files:** `cnc_<feature>.py` for core/common, `<machine_type>.py` for machine-specific
- **Config sections:** `[cnc_<feature>]` for core/common, `[<machine_type>]` for machine-specific
- **G-code commands:** Use standard G/M codes where they exist (G54, M3, etc.), prefix custom commands with machine type
- **Python classes:** PascalCase, matching Klipper convention (e.g., `CncWorkCoordinates`, `WaterjetController`)
- **Config keys:** snake_case, matching Klipper convention (e.g., `pierce_delay`, `door_switch_pin`)

## Safety / Warnings

- **Interlock modules are safety-critical** â€” they must fail-safe (deny motion/output if sensor state is unknown)
- **Output sequencing must be deterministic** â€” timed sequences must complete or abort cleanly, never leave outputs in partial state
- **Machine-specific safety modules must document all interlocks** with explicit "what happens if this sensor fails" analysis
- **Never assume hardware state** â€” always read/confirm before acting (e.g., confirm door closed, not just "door not reported open")
- **Watchdog integration** â€” modules that control dangerous outputs (laser, plasma, waterjet HP pump) must register with Klipper's watchdog so outputs disable on host disconnect
