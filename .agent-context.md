## Known Issues / Blockers

- **Feed hold may require Klipper core changes** — true mid-move pause needs motion planner hooks. Need to investigate if this can be done purely in extras/ or if a core PR is needed.
- **Cutter compensation (G41/G42) requires look-ahead** — complex, may also need core changes. Low priority since CAM handles kerf comp for waterjet/plasma.
- **Klipper's M3/M4/M5 is PWM-only** — stock `[pwm_tool]` maps spindle commands to PWM duty cycle. We need digital on/off M3/M5 for waterjet valves and plasma arc.

## Decisions & Context

### Why a module library, not a fork?
Klipper's `klippy/extras/` system is the intended extension mechanism. Modules only load when referenced in `printer.cfg` — zero bloat for users who don't need CNC features. This approach:
- Works with stock Klipper (no fork to maintain)
- Distributable via Moonraker's update manager
- Can be upstreamed to Klipper if maintainer accepts PRs
- Won't go stale when Klipper updates (no merge conflicts)

### Why one repo, not one per module?
Modules are interdependent (waterjet depends on output_sequencer depends on interlock). One repo = one version = no compatibility matrix. This matches how klippy/extras/ itself works — all in one tree, load only what's configured.

### Architecture: 4-tier module system
1. **core/** — Every CNC needs these (WCS, feed hold, overrides, probing)
2. **common/** — Most machines need, configure per type (spindle, coolant, tool table, interlocks, output sequencer)
3. **machine/** — Shared by 2-3 machine classes (laser PWM, pierce cycle, THC, gap servo)
4. **machines/** — Machine-specific (waterjet, plasma, mill, EDM, lathe, PnP)

### Key design insight: output_sequencer is the universal module
Nearly every CNC type needs "turn on output A, wait X ms, turn on output B, wait Y ms, begin motion." One parameterized module serves waterjet pierce, plasma arc start, laser gas pre-flow, mill spindle spin-up, and foam wire warm-up.

### Key design insight: interlock is the universal safety module
Every machine has "check input X before allowing output Y." One configurable module serves door interlocks, flow sensors, pressure confirms, chuck clamps, arc OK signals, etc.

### Machine types analyzed
Full requirements analysis completed for: waterjet, plasma, CO2 laser, fiber laser, diode laser, mill/router, wire EDM, sinker EDM, lathe, drag knife, foam cutter, pick-and-place. See architecture.md for per-machine breakdown.

### Primary development target: Wazer waterjet
We have a fully reverse-engineered Wazer desktop waterjet cutter:
- MCU: LPC1769 (Klipper-supported)
- Complete pin map documented (see bharrison6/Wazer repo)
- Full state machine, I/O sequences, safety interlocks mapped from firmware decompilation
- Stock Smoothieware diffed: only 1 base function modified + 183-function waterjet module (51KB)
- This is our test platform for validating the module architecture

### Klipper stack reference
```
Browser (Mainsail/Fluidd) → Moonraker (API) → Klippy (host/Python) → MCU firmware (C on LPC1769)
                                                  ↑
                                          Our modules live here
                                         (klippy/extras/*.py)
```
